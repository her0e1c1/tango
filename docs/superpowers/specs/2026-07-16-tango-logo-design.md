# Tango Logo Design

## Context

Issue #227 requests an official Tango logo in SVG, versions suitable for the app and README, and a README update. Tango is a flashcard application for programmers. Its current Calm Focus visual system uses indigo, teal, near-white, and rounded surfaces.

The existing mark already suggests stacked cards and a `T`, but it contains a decorative dot that becomes unclear at favicon size. The current horizontal treatment also repeats the initial visually: the `T` inside the mark is followed by the complete word `tango`.

## Goals

- Make the flashcard metaphor recognizable from 16px upward.
- Preserve the calm, focused personality of the current UI while retaining some friendliness.
- Provide self-contained SVG sources for the standalone mark and horizontal logo.
- Use the selected logo consistently in the app, PWA assets, browser chrome, and README.
- Keep the current `Logo` component API and interaction accessibility.

## Explored Directions

Three mark directions were compared:

1. **Stacked T** — two offset cards with a `T` on the front card.
2. **Flip Notes** — stacked notes with text lines and a flip/check detail.
3. **Front / Back** — a split card representing the question-and-answer rhythm.

Stacked T was selected because it communicates flashcards with the fewest shapes and remains clearest at small sizes.

Three horizontal structures were then compared:

1. Use the mark as the initial `T` and follow it with `ango`.
2. Put the complete `TANGO` name inside a wide flashcard badge.
3. Use a wordmark without the mark.

The mark-as-`T` structure was selected to avoid reading the current lockup as “T tango.” Four variations of that structure were reviewed. **W1-D, Card trail**, was selected: the mark leads into `ango`, which sits on a card surface extending behind the mark.

## Selected Visual Design

### Standalone mark

The mark uses a 64 by 64 view box:

- an indigo rounded-square tile (`#4f63b8`);
- a teal back card (`#2f7f78`);
- a near-white front card (`#f8fafc`);
- a centered indigo `T` cut into the front card.

The decorative dot from the existing asset is removed. This leaves only the details that remain useful at 16px, 32px, and 64px. The mark keeps the same fixed colors on light and dark backgrounds so the app icon remains recognizable.

### Horizontal Card trail logo

The mark serves as the first letter of `tango`. The remaining outlined letters `ango` sit on a rounded card surface that begins behind the mark. A short teal offset beneath the card suggests the second card in a stack.

`public/tango-logo.svg` is the canonical geometry for every horizontal use. It has a `216 64` view box and uses these layers in back-to-front order:

1. A teal rear card at `x=36`, `y=14`, `width=176`, `height=42`, and `rx=8`.
2. A front card at `x=32`, `y=8`, `width=180`, `height=44`, and `rx=8`.
3. The custom `ango` path group spanning `x=78` through `x=200`, with its main letter bodies centered near `y=34` and the `g` descender reaching `y=54`; it uses a `4.5` unit stroke, round caps, and round joins.
4. The approved 64 by 64 standalone mark at `x=0`, `y=0`, above the leading edge of both cards.

The `ango` lettering is four canonical open paths, stored without transforms:

```text
a: M97 25v19 M97 34c0-6-4-10-9-10s-10 4-10 10 4 10 10 10 9-4 9-10
n: M112 44V25m0 8c2-6 6-9 11-9 6 0 9 4 9 10v10
g: M165 25v17c0 8-4 12-11 12-5 0-9-2-11-5 M165 34c0-6-4-10-10-10s-10 4-10 10 4 10 10 10 10-4 10-10
o: M189 24c-7 0-11 4-11 10s4 10 11 10 11-4 11-10-4-10-11-10z
```

The lettering therefore uses SVG paths rather than a live font. This keeps rendering stable across the app, GitHub, and exported assets. Light and dark variants share all coordinates and path data; only these colors differ:

- the light variant uses front card `#f8fafc`, lettering `#202936`, and rear card `#2f7f78`;
- the dark variant uses front card `#182231`, lettering `#edf2f7`, and rear card `#2f7f78`.

Both variants reuse the same fixed-color standalone mark. `public/tango-logo-dark.svg` is generated by copying the canonical SVG and changing only the front-card and lettering colors listed above.

## Repository Integration

### Assets

- Refine `public/tango-mark.svg` as the scalable standalone source.
- Add `public/tango-logo.svg` for the light Card trail lockup.
- Add `public/tango-logo-dark.svg` for the dark Card trail lockup.
- Regenerate `public/logo192.png`, `public/logo512.png`, and `public/favicon.ico` from the standalone mark.
- Keep `public/manifest.json` and `index.html` pointing to the standalone mark and derived PWA assets.

Rejected concepts are documented here but are not added as production assets.

### App

Keep the public props of `Logo` unchanged: `onClick`, `className`, and `markOnly`.

- `markOnly` continues to render the standalone mark with a screen-reader-only `tango` label.
- The normal mode renders the canonical light and dark Card trail SVG assets at `108` by `32`. Tailwind light/dark visibility classes select the matching asset, so the React component does not duplicate logo geometry.
- The clickable logo keeps its existing button interaction hook, keyboard behavior, and accessible name `tango`.
- Storybook retains explicit wordmark, mark-only, light, and dark review states.

Both visible logo images are decorative (`alt=""` and `aria-hidden="true"`); one screen-reader-only `tango` label supplies the accessible name in both normal and mark-only modes. The root still accepts caller classes. Theme visibility classes do not encode palette values, and the SVG assets provide the exact same geometry outside React.

### README

Replace the Markdown `# Tango` heading with one HTML `<h1>` containing a `<picture>`. The picture uses a dark-scheme `<source>` for `public/tango-logo-dark.svg` and a fallback `<img>` for `public/tango-logo.svg`. The fallback image has `alt="Tango"`, so the heading retains an accessible document title without a second visible or hidden title.

## Verification

Extend the existing PWA identity contract tests to verify:

- the refined mark has the expected view box and palette and no decorative dot;
- both horizontal SVG variants exist, contain accessible titles, share geometry, and use outlined paths rather than live `<text>`;
- README references the light and dark horizontal assets with appropriate alternative text;
- the raster icons remain valid at 192px and 512px, the favicon remains a valid 64px icon, and their expected hashes match the regenerated files;
- the `Logo` component uses the horizontal assets at 108 by 32, exposes one accessible name `tango`, preserves click and keyboard behavior, and keeps `markOnly` accessible;
- Storybook retains light and dark review states without raw color utilities.

Before the PR is opened, run `make check`. Also render or inspect the final SVGs and raster exports at 16px, 32px, 64px, README scale, and on light and dark backgrounds.

## Failure Handling

Asset generation must fail visibly if the SVG cannot be rasterized or if an output has the wrong dimensions. Contract tests validate file structure and hashes so stale or partially generated outputs cannot pass silently. If visual inspection reveals loss of clarity at a target size, adjust the SVG source and regenerate every derived asset together.

## Non-goals

- Redesigning the broader Calm Focus palette or typography system.
- Changing navigation, layout, or the `Logo` component API.
- Shipping rejected logo concepts as production assets.
- Adding a runtime image-generation dependency.
